# AGENTS.md - 执行协议

## 0. 五条硬规则
1. 每次进入计划阶段前，或执行架构设计 / 技术选型 / 方案对比 / 中大型修改 / 复杂调试等复杂任务前：必须先调用 `$github-kb` 做至少一轮检索，并输出可追溯证据。
2. 每次代码修改 / 重构 / 调试后：必须调用 `$git-md-micro-commit` 进行存档（按 Markdown-Tagged Micro-Commit 结构生成并提交）；不询问用户。
3. 每次最终答复后：默认调用 `$codex-notifier` 发送弹窗 + 声音；用户要求“关闭响铃”只关声音，要求“关闭通知”则全部关闭。
4. 每次执行任何代码修改前：先阅读当前 Git 仓库至少 `20` 条提交记录；若历史上下文不足以支撑判断，必须继续阅读更多提交记录再执行修改。
5. 每次任务只能进行一个功能的开发/改动；若发现需求跨越多个功能，需拆分为多个独立任务并逐个完成。
6. 每次任务中如果不能完全通过完整构建+测试回归，不能将任务状态改为true，如果需要用户手动完成的测试任务，需要先指导用户完成，得到确认回复后才能将任务状态改为true。

## 1. 标准执行顺序
- 计划流程：`$github-kb`（默认 `1` 轮，证据不足补 `1` 轮） -> 计划产出 -> 最终答复 -> `$codex-notifier`
- 普通改码流程：读取至少 `20` 条提交记录 -> 代码变更 -> `$git-md-micro-commit` -> 最终答复 -> `$codex-notifier`
- 复杂改码流程：读取至少 `20` 条提交记录 -> `$github-kb`（默认 `1` 轮，证据不足补 `1` 轮） -> 代码变更 -> `$git-md-micro-commit` -> 最终答复 -> `$codex-notifier`

## 2. `$github-kb` 高质量代码参考规则
1. 状态检查
- 仅当任一 `gh` 命令出现认证 / 主机 / 网络错误时，触发复检：`gh auth status`；仍异常再执行：`gh auth login`
2. 检索轮次与预算（压缩）
- 默认仅执行 `1` 轮最小必要检索（建议 `limit 5-10`）
- 若未满足“至少 `2` 个独立来源链接 + 可迁移规则摘要”，仅允许补 `1` 轮检索
- 超过 `2` 轮检索需用户明确要求
- 同一任务内复用已验证证据，禁止为同一结论重复同类检索
3. 三要素门槛（采纳参考实现前必须检查）
- 活跃度：候选仓库近 `3` 年内存在更新（以 `pushedAt/updatedAt` 为准）
- 工程质量信号：至少满足以下 `2` 项
  - 存在 CI 流程（如 `.github/workflows`）
  - 存在可识别测试信号（测试目录、测试配置或测试工作流）
  - 存在协作质量信号（`CONTRIBUTING`、`CODEOWNERS`、PR 模板）
- 许可证风险：默认提示许可证类型与风险，尤其 GPL/AGPL
4. 证据输出最小要求
- 至少 `2` 个独立来源链接（不同基础域名或不同仓库主体）
- 给出可迁移规则摘要，不只罗列链接
- 若证据不足：仅输出 `临时建议（Low confidence）+ 缺失证据清单`
5. 降级策略
- `gh` 不可用或未认证时，必须明确报错并提示 `gh --version` / `gh auth status` / `gh auth login`
- 降级后不做强结论，只给临时建议与待补证据项

## 3. Python 环境规则
- 所有 Python / pip 命令必须在项目内由 `uv` 创建并管理的虚拟环境中执行（默认 `.venv`，例如通过 `uv venv` 初始化）
- 禁止使用系统 Python、全局 pip，或任何非 `uv` 管理的虚拟环境

## 4. 兜底
- `$git-md-micro-commit` 不可用：按其模板手动撰写结构化 Commit Message 并执行 `git commit`；若当前目录非 Git 仓库，在最终答复明确“未完成存档（非 Git 仓库）”
- `$codex-notifier` 不可用：在最终答复中明确写“通知未发送（skill 不可用）”
- `$github-kb` 不可用：在最终答复中明确写“证据检索未完成（github-kb 不可用）”，并附 `gh` 修复指引
