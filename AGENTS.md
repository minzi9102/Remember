# AGENTS.md - 执行协议

## 0. 五条硬规则
1. 每次代码修改 / 重构 / 调试后：必须调用 `$git-md-micro-commit` 进行存档（按 Markdown-Tagged Micro-Commit 结构生成并提交）；不询问用户。
2. 每次最终答复后：默认调用 `$codex-notifier` 发送弹窗 + 声音；用户要求“关闭响铃”只关声音，要求“关闭通知”则全部关闭。
3. 架构设计 / 技术选型 / 方案对比 / 中大型修改或复杂调试：先调用 `$tech-evidence-research` 做至少一轮外部检索，再给方案或开改代码。
4. 每次执行任何代码修改前：先阅读当前 Git 仓库至少 `20` 条提交记录；若历史上下文不足以支撑判断，必须继续阅读更多提交记录再执行修改。
5. 每次任务只能进行一个功能的开发/改动；若发现需求跨越多个功能，需拆分为多个独立任务并逐个完成。

## 1. 标准执行顺序
`$tech-evidence-research`（如触发） -> 代码变更 -> `$git-md-micro-commit` -> 最终答复 -> `$codex-notifier`

## 2. `$tech-evidence-research` 证据规则
1. 首行必须输出：`[Skill] 已启用：$tech-evidence-research`
2. 固定流程：`同类发现 -> 方案证据 -> 交叉验证 -> 评分推荐`
3. 每个候选方案至少 2 个独立外部案例（按基础域名判独立）
4. 证据不足时只允许：`临时建议（Low confidence）+ 缺失证据清单`
5. 来源冲突必须保留并给出主判断理由
6. 默认提示许可证风险（尤其 GPL/AGPL）
7. 固定输出结构：
`[Skill] 已启用：$tech-evidence-research -> 竞品拆解 -> 可迁移点 -> 3个候选方案对比 -> 推荐方案 -> 证据表 -> 风险与缺口`

## 3. 检索默认参数（可覆盖）
- 检索预算：`5-10`
- 首轮时延目标：`20-60s`
- 会话缓存：`15min`
- 案例优先时间窗：近 `3` 年（资讯默认近 `30` 天）
- 输出语言：中文优先，自动扩展英文检索词
- 模式：默认 `web-only`，MCP 按需接入

## 4. Python 环境规则
- 所有 Python / pip 命令必须在项目内由 `uv` 创建并管理的虚拟环境中执行（默认 `.venv`，例如通过 `uv venv` 初始化）
- 禁止使用系统 Python、全局 pip，或任何非 `uv` 管理的虚拟环境

## 5. 兜底
- `$git-md-micro-commit` 不可用：按其模板手动撰写结构化 Commit Message 并执行 `git commit`；若当前目录非 Git 仓库，在最终答复明确“未完成存档（非 Git 仓库）”
- `$codex-notifier` 不可用：在最终答复中明确写“通知未发送（skill 不可用）”
