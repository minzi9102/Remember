# EXECUTION RULES

## 1. 黑盒原则
1. 允许使用：应用 UI、配置文件、日志输出、SQLite Browser。
2. 禁止依赖：源码阅读、内部函数名、未公开调试接口。
3. 若步骤依赖“交付方提供的启动命令/测试账号/测试数据”，按测试包文档执行。

## 2. 环境矩阵
| 环境ID | OS | 运行模式 | 数据源 |
|---|---|---|---|
| ENV-SQLITE | Windows/macOS | `sqlite_only` | 本地 SQLite |

## 3. 门禁执行流程
1. 按阶段顺序执行：Phase 1 -> Phase 2 -> Phase 3 -> Phase 4 -> Phase 5。
2. 每个子任务按顺序执行四条用例：`VG-PASS` -> `IG-PASS` -> `VG-FAIL` -> `IG-FAIL`。
3. 每条用例执行后立即上传证据并记录结果（PASS/FAIL/BLOCKED）。
4. 当 FAIL 用例未触发预期保护时，记录为缺陷并标记严重度。

## 4. 统一结果判定
- PASS：步骤完成且预期结果全部满足。
- FAIL：步骤完成但预期结果不满足。
- BLOCKED：因环境或前置条件不足无法执行。

## 5. 失败分级标准
| 等级 | 判定标准 | 处理要求 |
|---|---|---|
| 阻断 | 关键路径不可用/数据一致性破坏/无法发布 | 立即停止后续发布门禁 |
| 高 | 核心功能异常但存在临时绕过 | 24 小时内给出修复方案 |
| 中 | 非核心流程偏差或提示不一致 | 迭代内修复 |
| 低 | 文案或轻微视觉偏差，不影响主流程 | 可排期修复 |

## 6. 证据提交最小要求
1. 视觉证据：截图或录屏，必须包含时间戳与关键 UI 状态。
2. 交互证据：操作记录（文本）+ 日志片段或数据库查询结果。
3. 每条用例至少 2 份证据（视觉 1 + 交互 1）。

## 7. 发布门禁条件
1. 所有阻断级问题必须为 0。
2. 所有高优先级问题必须具备关闭结论（已修复或风险豁免签字）。
3. `MASTER-TRACE-MATRIX.md` 状态列必须完整。
