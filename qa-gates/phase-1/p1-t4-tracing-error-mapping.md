# P1-T4 tracing 与错误码映射 门禁测试方案

## 目标
- 验证日志结构化输出和错误码可观测性。
- 本文件聚焦：错误码可视化

## 前置条件
1. 已获取本版本可执行程序、启动命令与测试账号（如需登录）。
2. 可访问目标配置文件（config.toml）并允许重启应用。
3. 可查看应用日志目录；如涉及数据一致性，已安装 SQLite Browser 与 psql。
4. 所有步骤均按黑盒执行，不阅读源码。

## 测试数据
| 数据项 | 建议值 |
|---|---|
| 运行模式 | sqlite_only |
| 样例 Series | Inbox, Project-A, Archive-Me |
| 样例 Commit | first-note, follow-up-note, rollback-check |
| 异常输入 | 空字符串、超长文本(2000字)、非法模式值 |

## 视觉门禁（2例）
### P1-T4-VG-PASS
- 用例目的：验证 错误码可视化 在正常路径下可见且稳定。
- 执行步骤：
  1. 使用目标环境启动应用并进入与本子任务相关的页面/状态。
  2. 执行本子任务主流程（按 ROADMAP 对应子任务操作）。
  3. 截图关键界面（操作前/操作后至少各 1 张）。
  4. 重点核对：日志字段完整可检索。
- 预期结果：
  1. 关键 UI 元素完整可见，无错位、无空白块、无异常闪烁。
  2. 成功态反馈清晰（状态、提示、排序或标识与预期一致）。
- 必交证据：P1-T4-VG-PASS_*.png + P1-T4-VG-PASS_*.txt

### P1-T4-VG-FAIL
- 用例目的：验证反向条件下有明确的可视保护（错误提示/降级态/阻断态）。
- 执行步骤：
  1. 人为设置反向条件（如非法配置、目标依赖不可达、输入不合法）。
  2. 重复主流程并观察 UI 反馈。
  3. 重点核对：错误码显示准确可读。
- 预期结果：
  1. 出现明确可读的失败提示，不得静默失败。
  2. 失败场景下界面不崩溃，关键信息仍可访问。
- 必交证据：P1-T4-VG-FAIL_*.png + P1-T4-VG-FAIL_*.txt

## 交互门禁（2例）
### P1-T4-IG-PASS
- 用例目的：验证本子任务正向交互可形成闭环。
- 执行步骤：
  1. 按本子任务定义完成一次完整操作链路。
  2. 记录每一步输入动作（键盘/鼠标/快捷键）与系统反馈。
  3. 核对：正常+异常操作均写入日志。
- 预期结果：
  1. 操作过程无卡死、无误触发、无不可恢复状态。
  2. 结果与预期一致，且可被日志/数据库结果佐证。
- 必交证据：P1-T4-IG-PASS_*.mp4 + P1-T4-IG-PASS_*.txt

### P1-T4-IG-FAIL
- 用例目的：验证反向交互被正确拦截且系统保持稳定。
- 执行步骤：
  1. 执行一次非法或冲突操作（如无效输入、依赖断开、重复触发）。
  2. 观察并记录系统拦截行为和恢复路径。
  3. 再执行一次合法操作确认系统可恢复，并核对：日志异常时应用仍可运行并提示。
- 预期结果：
  1. 非法交互被拒绝并返回明确提示/错误码。
  2. 系统不崩溃，后续合法操作可继续完成。
- 必交证据：P1-T4-IG-FAIL_*.mp4 + P1-T4-IG-FAIL_*.txt

## 证据要求
1. 命名必须遵循 qa-gates/EVIDENCE-NAMING.md。
2. 每条用例至少提交 1 份视觉证据 + 1 份交互证据。
3. 文本证据需包含：执行环境、步骤编号、实际结果、结论。
4. 若涉及数据库一致性，附 SQLite 与 PostgreSQL 查询截图或导出文本。

## 通过标准
1. P1-T4-VG-PASS 与 P1-T4-IG-PASS 均通过。
2. P1-T4-VG-FAIL 与 P1-T4-IG-FAIL 均能触发预期保护。
3. 证据完整可追溯，结论可被第三方复验。

## 失败分级
| 等级 | 判定标准 | 示例 |
|---|---|---|
| 阻断 | 主路径不可用或一致性破坏 | 无法提交、双库数据冲突、应用崩溃 |
| 高 | 核心行为错误但可临时绕过 | 错误码不准、排序异常、模式切换失效 |
| 中 | 非核心流程偏差 | 提示文案不一致、局部 UI 状态错误 |
| 低 | 轻微视觉问题 | 对齐偏差、图标不一致 |
