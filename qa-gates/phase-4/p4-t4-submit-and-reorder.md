# P4-T4 提交后最左置顶与摘录刷新 门禁测试方案

## 目标
- 验证 Commit 提交后目标 Series 立即移动到主轨道最左并刷新最新摘录。
- 本文件聚焦：提交后的排序与卡片信息刷新。

## 前置条件
1. 已获取本版本可执行程序与启动命令。
2. 可访问目标配置文件（`config.toml`）并允许重启应用。
3. 可查看应用日志目录；如涉及一致性验证，可访问 SQLite 查询工具。
4. 所有步骤按黑盒执行，不阅读源码。

## 测试数据
| 数据项 | 建议值 |
|---|---|
| 运行模式 | sqlite_only |
| 样例 Series | Inbox, Project-A, Archive-Me |
| 样例 Commit | first-note, follow-up-note, rollback-check |
| 异常输入 | 空字符串、超长文本(2000字) |

## 视觉门禁（2例）
### P4-T4-VG-PASS
- 用例目的：验证提交后置顶与摘录刷新在正常路径可见。
- 执行步骤：
  1. 在主轨道选择一个非最左 Series。
  2. 直接输入触发附着草稿并提交。
  3. 截图提交前后轨道顺序与卡片摘录。
- 预期结果：
  1. 提交后目标 Series 立即到达最左位置。
  2. 卡片摘录更新为最新 Commit 内容。
- 必交证据：`P4-T4-VG-PASS_*.png` + `P4-T4-VG-PASS_*.txt`

### P4-T4-VG-FAIL
- 用例目的：验证提交失败路径下有明确保护反馈。
- 执行步骤：
  1. 构造空提交或非法提交。
  2. 观察是否出现错误提示与草稿保留策略。
  3. 检查轨道顺序未发生错误重排。
- 预期结果：
  1. 失败场景有明确错误反馈，不得静默失败。
  2. 不会出现错误置顶或摘录污染。
- 必交证据：`P4-T4-VG-FAIL_*.png` + `P4-T4-VG-FAIL_*.txt`

## 交互门禁（2例）
### P4-T4-IG-PASS
- 用例目的：验证多 Series 连续提交时排序规则稳定。
- 执行步骤：
  1. 依次对多个 Series 提交 Commit。
  2. 每次提交后记录最左卡片与高亮变化。
  3. 核对最新提交项始终位于最左。
- 预期结果：
  1. 每次提交都触发即时最左置顶。
  2. 摘录与数据库最新记录一致。
- 必交证据：`P4-T4-IG-PASS_*.mp4` + `P4-T4-IG-PASS_*.txt`

### P4-T4-IG-FAIL
- 用例目的：验证反向交互被拦截且系统保持稳定。
- 执行步骤：
  1. 触发空输入提交。
  2. 触发 pending 状态下重复提交。
  3. 再执行一次合法提交确认恢复。
- 预期结果：
  1. 非法提交被拒绝并返回明确错误。
  2. 系统可恢复，后续合法提交正常置顶。
- 必交证据：`P4-T4-IG-FAIL_*.mp4` + `P4-T4-IG-FAIL_*.txt`

## 证据要求
1. 命名遵循 `qa-gates/EVIDENCE-NAMING.md`。
2. 每条用例至少提交 1 份视觉证据 + 1 份交互证据。
3. 文本证据需包含：执行环境、步骤编号、实际结果、结论。
4. 涉及数据一致性时，附 SQLite 查询截图或导出文本。

## 通过标准
1. `P4-T4-VG-PASS` 与 `P4-T4-IG-PASS` 均通过。
2. `P4-T4-VG-FAIL` 与 `P4-T4-IG-FAIL` 能触发预期保护。
3. 证据完整可追溯，结论可被第三方复验。

## 失败分级
| 等级 | 判定标准 | 示例 |
|---|---|---|
| 阻断 | 主路径不可用或一致性破坏 | 提交成功但未置顶、应用崩溃 |
| 高 | 核心行为错误但可临时绕过 | 摘录未刷新、顺序错乱 |
| 中 | 非核心流程偏差 | 提示文案不一致、状态延迟 |
| 低 | 轻微视觉问题 | 高亮样式异常、卡片间距偏差 |
